# Project Map Update Rules

This note governs how the Obsidian project map is kept useful as it grows.

## Node Rule

Every project node must have:

- `type: project-node`
- stable `node_id`
- `surface`
- `level`
- `parent_node` for level-2 and deeper nodes
- `state`
- `confidence`
- `last_verified`
- `update_rule`
- `state_source`
- `source_refs`
- `dfd_refs`
- `proof_refs`
- tags for `ta3000/project-node`, `ta3000/project-graph`, `surface/*`,
  `level/*`, and `state/*`

## Depth Rule

- Level 1 is for stable map areas: shell, product plane, data, research,
  runtime, execution, contracts, and the map itself.
- Level 2 is for capability contours that a human can reason about without
  opening code.
- Level 3 is allowed only when one level-2 contour becomes too dense to answer
  "what depends on what?"

Do not add one node per source file. Source files belong in source docs, not in
the visual map.

## Link Rule

Use graph links for durable relationships only:

- parent contains child;
- output feeds next capability;
- contract governs consumer;
- policy constrains runtime behavior;
- item points to the node it affects.

Do not link a node merely because two notes mention the same topic.

## Reference Rule

Every node must tie back to real project material:

- `source_refs`: canonical docs, runbooks, schemas, or config that define the
  node.
- `dfd_refs`: DFD page(s) that show the node's process/data-flow context.
- `proof_refs`: validators, tests, scripts, schemas, or proof surfaces that can
  substantiate the node's current state.

Use references for traceability. Use wikilinks only when the relationship
should be visible as a graph edge.

## Visual Rule

`project-cockpit.html` is the primary readable visual map. It is generated from
the markdown nodes and items by `scripts/build_project_cockpit.py`.

`Project Map Overview` is the compact DFD-like orientation map. It must include
a Mermaid overview and stay aligned with the generated cockpit's level-1 and
level-2 structure.

Global Graph is secondary. Obsidian may rewrite `.obsidian/graph.json` while a
graph tab is open, so graph runtime settings are not the durable project-map
contract.

## State Rule

Node `state` is manual but evidence-backed. It describes the current confidence
in that project area, not the urgency of every linked question:

- `ok`: current source docs and proof do not show open attention.
- `attention`: the area needs review or active follow-up.
- `blocked`: cannot move without missing external action or proof.
- `unknown`: not verified enough to claim status.

The generated cockpit computes a separate roll-up status for large blocks:

- `stable`: no active exceptions found in the block.
- `watch`: only low-priority active items are present.
- `review`: the block has attention/review work but no immediate decision.
- `unknown`: at least one area is not verified enough.
- `decision`: user attention or P1 items exist.
- `blocked`: blocked node or P0 item exists.

This keeps large blocks readable when many low-priority questions exist.

## Priority Rule

Project items must carry `priority` separately from `severity`:

- `p0`: blocks current progress.
- `p1`: needs a user/agent decision soon.
- `p2`: should be reviewed, but does not block current work.
- `p3`: watch/backlog item.

`severity` remains impact. `priority` is operating order.

## Origin Rule

Every project item must declare where it came from:

- `origin_kind`: `manual`, `manual-seed`, `sync-node-state`, or
  `sync-task-blocker`.
- `origin_refs`: source note, task note, node note, report, or decision document
  that justifies the item.
- `sync_managed: true` for items generated by
  `scripts/sync_project_map_items.py`.

Manual prototype items that no longer represent current project reality must be
set to `status: dropped`; they can remain as history, but they must not pollute
the active priority queue.

Generated current-state items are not edited by hand. Fix the source signal
instead: update node state or add the missing evidence.

`docs/tasks/active` is legacy diagnostic input, not an authoritative source for
the current map. Task notes were created across separate worktrees and were not
reliably merged back to main, so their blocker sections must not enter the
default cockpit queue. Use `scripts/sync_project_map_items.py
--include-legacy-task-notes` only for an explicit audit of old task artifacts.

If old `sync-task-blocker` item cards already exist, archive them under
`docs/archive/historical-project-map-items/` instead of leaving them in the
active item folder. Keeping them under `state/items/` can make historical
blockers appear current even when their source task notes have been retired.

## Candidate Rule

Memory-recovered signals are candidates, not active project items. Use
`scripts/collect_project_map_candidates.py` to refresh the candidate inbox from
MemPalace. Promote a candidate only after current repo evidence, runtime
artifact evidence, or an explicit user/agent decision confirms that the signal
is still live.

The candidate report must not affect cockpit roll-up status directly.

## Update Triggers

Update this map when any of these change:

- canonical architecture map or repository surface split;
- product-plane implemented-reality status source;
- contract inventory or compatibility policy;
- native runtime ownership;
- accepted data root, baseline pin, or route ownership;
- research primary path, registry tables, or campaign route;
- runtime persistence/API/signal lifecycle;
- live decision policy or real broker closure;
- delivery gate, agent routing, or validation artifact policy.

## Review Cadence

- Update nodes in the same change that changes their source-of-truth docs.
- If a source doc changed and the matching node was not reviewed, set
  `state: unknown` or create a project item.
- Prefer one new item over speculative node edits when the state is unclear.
- Give every new item a `priority` and tag it as `priority/*`.
- Run `python scripts/sync_project_map_items.py` before rebuilding the cockpit.
  Do not include legacy task notes in normal map maintenance.
- Run `python scripts/validate_project_map.py` after project-map changes.
- Run `python scripts/build_project_cockpit.py` after project-map changes and
  `python scripts/build_project_cockpit.py --check` in closeout.
